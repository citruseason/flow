## Agenda

Analyze gaps between the OpenAI Harness Engineering article and Flow's current implementation, then define scope for improvements.

## Research Summary

16 gaps identified across 4 categories:
- **A. Knowledge Base & Documentation (6):** Core beliefs, product sense, references, domain docs, architecture doc, auto-generated docs
- **B. Execution & Autonomy (5):** Execution plans, background agents, full lifecycle, self-review loop, merge philosophy
- **C. Tooling & Observability (3):** Agent-facing observability, bug reproduction/video, per-worktree app boot
- **D. Lint & Quality (3):** Structural tests, agent-optimized lint errors, per-domain quality grades

## Dialogue

### User Direction
User identified two core concerns:
1. Web/app DESIGN management is missing
2. Agents lack elements to quickly understand user projects with minimal context

### CORE Domain Documents

**Decision:** Introduce CORE domain documents at `harness/` root level.

- **Always generated:** PRODUCT.md, SECURITY.md
- **Conditionally generated** (based on project analysis): DESIGN.md (UI/UX), FRONTEND.md, BACKEND.md, BATCH.md, OBSERVABILITY.md, INFRA.md, DATA.md, etc.
- **Scope:** Product-level, NOT per-app/per-package. Even in monorepos, CORE represents the whole product.
- `/harness-init` detects project type and generates relevant domain documents only.

### CORE as Lint Skill Source (Decision: b)

**Decision:** CORE domain documents serve as the upstream source for lint skills.

- CORE defines "why and what" (design decisions and principles)
- Lint skills reference CORE to generate "how" (mechanical verification rules)
- When CORE changes, lint rules update accordingly
- This matches the original article: docs as "single source of truth", linters enforce mechanically

### CORE Update Trigger (Decision: b)

**Decision:** CORE updates are triggered at merge time, not at meeting/design-doc completion.

- Rationale: Only realized facts should enter CORE, not plans or documents that haven't materialized in code yet
- Implementation: git hook or Claude Code hook detects merge event

### Agent Entry Point

**Decision:** Add harness map section to user project's CLAUDE.md automatically during `/harness-init`.

- CLAUDE.md acts as the ~100-line map/table of contents (matching article's AGENTS.md pattern)
- Points to harness/ domain documents for deeper context
- Enables progressive disclosure: agents start from CLAUDE.md, navigate to specific domain docs as needed

### File Structure (Decision: harness/ root, not harness/core/)

**Decision:** CORE domain documents live at `harness/` root, not in a `core/` subdirectory.

```
project-root/
├── CLAUDE.md              (harness section auto-added)
└── harness/
    ├── PRODUCT.md         (always)
    ├── SECURITY.md        (always)
    ├── DESIGN.md          (conditional)
    ├── FRONTEND.md        (conditional)
    ├── BACKEND.md         (conditional)
    ├── quality-score.md
    ├── tech-debt.md
    ├── kanban.json
    ├── references/
    └── topics/
```

### Legacy File Cleanup

**Decision:** Remove files that are absorbed into domain documents:

- `index.md` → removed (CLAUDE.md takes over map role)
- `golden-rules.md` → rules distributed to relevant domain documents, then removed
- `observability.md` → absorbed into FRONTEND.md / BACKEND.md etc., then removed
- `quality-score.md` → kept (cross-cutting operational document)
- `tech-debt.md` → kept (cross-cutting operational document)

### Scope Decision

User confirmed scope limited to:
1. CORE domain documents introduction
2. CLAUDE.md as agent entry point
3. Legacy file restructuring
4. CORE as lint skill upstream source
5. Merge-time CORE update trigger

Explicitly out of scope for this iteration:
- B1. Execution plans as first-class artifacts
- B2. Background/periodic agents (garbage collection)
- B4. Agent self-review loop (Ralph Wiggum Loop)
- D1. Structural tests (ArchUnit-style)
- D2. Agent-optimized lint error messages
- D3. Per-domain quality grades

### Reference Management (Added Post-Draft)

User identified that `harness/references/` is empty and external reference management is completely missing.

**Decision:** Add automated reference management to scope.
- `/harness-init` analyzes dependencies and auto-collects reference material
- llms.txt fetched directly when available; agent extracts from official docs when not
- Users can manually add references
- New dependencies added during development trigger automatic reference collection
- Removed dependencies trigger stale reference cleanup
